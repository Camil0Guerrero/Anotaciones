Arquitectura CSS

Técnicas que nos ayudan a organizar y mantener nuestro código ordenado, optimo y escalable

Una buena arquitectura CSS debe ser:

  1. Predecible -> Que el nombre de nuestra clase sea alusiva a los estilos que aplicara, ademas que haga solo lo que dice que hace, no que tenga estilos que no tienen nada que ver con el nombre
  2. Reutilizable -> Que podamos aplicarlo en distintos contextos y contenedores de manera rápida sin que afecte otros estilos
  3. Estable -> Debemos poder modificar o actualizar con facilidad, sin afectar otras reglas
  4. 4. Escalable -> Cuando una aplicación crece decimos que se vuelve escalable, en este sentido nuestro código debe ser sencillo de mantener, sin aumentar mas de lo necesario el grado de complejidad. Lo recomendado es hacer clases utilitarias, es decir que cada una de ellas se dedique a aplicar algunos estilos específicamente, ademas podemos tomar como referencia un diccionario a la hora de definir nuestras clases, que podemos consultarlo cada que necesitemos aplicar estilos

Debemos diseñar SISTEMAS, no PAGINAS

Con esta frase nos referimos a que cada que creamos un proyecto no tenemos que iniciar de 0 siempre, Divide y vencerás D.R.Y -> Don't repeat yourself

Los componentes deben tener las siguientes características:

  1. Cumplen una sola función. -> Es mejor que cada cosa cumpla una funcionalidad, si le asignamos muchas, podemos enredarnos a la hora de querer usarlo de nuevo en otro proyecto. Es mejor que especifiquemos y se especialicen en una sola función y luego podemos unir varios para crear un proyecto mas complejo
  2. Son independientes. -> Si lo quitamos no debo porque nuestra interfaz verse afecta, al igual que al agregar nuevos
  3. Son autocontenidos. -> No permiten que componentes externos afecten su contenido e igualmente no afectan el contenido de otros ademas hacer lo posible que nuestro contenido por pequeño que sea, se pueda comprender y usar asi se encuentre solo
  4. Son reutilizables. -> Poder usarlo una y otra ves, aunque en cada proyecto tenga sus particularidades, que tenga una estructura genérica y luego fijarnos en los detalles o partes especificas

Herramientas

Estas herramientas css para crear sistemas nos permiten organizar y nombrar el código de una mejor manera

No tenemos que si o si ocupar las metodologías para todos los proyectos, sin embargo si es bueno saberlas usar y practicar con ellas en uno que otro, lo mejor seria usar una y acostumbrarnos a usarla siempre, aunque también podríamos usar sus practicas pero a nuestro modo

También podríamos tomar cosas de cada metodología, no tenemos que hacer todo a cabalidad y tal cual nos sugieren

Tomar lo mejor de cada una y crear su propia metodología, al fin y al cabo estas son tan solo propuestas de otros desarrolladores

Todas estas metodologías tratan de llegar al mismo punto: tener nuestro código CSS mas legible, mas organizado y entendible. Al igual que para cuando nuestro proyecto escale que nuestro código CSS este listo para escalar sin ningún problema

BEM

mas popular
Bloque Elemento Modificador
  1. Bloque -> Es un componente con todas las características que este conlleva
  2. Elemento -> Es una pequeña parte de ese bloque principal, como por ejemplo un logo en una cabecera. Para nombrar estos elementos se hace uso de __
  3. Modificador -> Las funciones que nuestro elemento tendrá, por ejemplo, el hover. Igualmente debemos separarlo con __

SMACSS

Scalable and Modular Architecture For CSS

La mayor diferencia con bem es que vamos a separar en capas los estilos CSS, vamos a tener categorías de reglas, Por ejemplo:

  • Core -> los estilos base
  • layout -> Composición de las areas del contenido
  • Module -> Las reglas que tengan que ver con módulos, lo podemos ver como componentes
  • state -> Aquellas que tienen que ver con los estados, active, hover, target, etc...
  • theme -> Los temas de nuestro sitio web, por ejemplo, el modo dark y light
  • changing -> Reglas que generan cambios, como un menu de hamburguesa, que en mobile y desktop cambia

A esta metodología también se le conoce como 7 en 1. Por sus capas y los estilos personales, los particulares para cada proyecto

Atomic Design

Tiene como filosofía el dividir nuestros componentes tanto como podamos, y luego al unirlos podemos formas componentes mas complejos

por ejemplo:

  • Un átomo -> Es un input
  • Una molécula -> Un formulario, donde ya debemos agregar otros átomos
  • un organismo -> La cabecera, donde hay varias moléculas
